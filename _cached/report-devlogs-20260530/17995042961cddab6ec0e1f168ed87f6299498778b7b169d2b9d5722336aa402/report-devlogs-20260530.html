74: Core Serialization — Postcard Binary + Content Enum, Remove serde_json::Value

Problem

serde_json::Value was used as Fact.content type and in petgraph NodeWeight.properties. This created several constraints:

  • Deserialize_any barrier: Binary serde formats (postcard, bincode) cannot serialize serde_json::Value because it requires deserialize_any, which binary formats do not support. Any attempt to switch from JSON to binary would require changing the core data type.
  • WASM incompatibility: serde_json is heavy for WASM. Worse, the previous SQLite backend (rusqlite with bundled C lib) made WASM compilation impossible. Even after SQLite removal, serde_json in core types prevented pure Rust + WASM targets.
  • Format lock-in: JSON in the core model meant all persistent data (snapshots, flush archives) had to be JSON. Switching to Parquet, Arrow, or any future format would require full data migration.
  • Performance: serde_json::Value is a heavy enum (String, Number, Array, Object, Bool, Null). Properties access required v.as_str(), v.as_i64(), v.as_bool() pattern matching. String comparisons on Content::Text are simpler and faster.

Solution

Content enum

Replace serde_json::Value with a minimal enum:

pub enum Content {
    Text(String),
    Blob(Vec<u8>),
}

Text holds plain strings or JSON strings as raw bytes. Blob holds binary data (images, serialized protobuf, etc.). The enum is trivially serializable by any serde-based format — postcard sees Text as (tag=0, len, bytes) and Blob as (tag=1, len, bytes).

Key design decisions:

Decision Rationale
No JsonString variant Text vs JsonString distinction at storage level is unnecessary. JSON parseability is a consumer concern.
From<&str> / From<String> → Content::Text Ergonomics: "text".into() works for Fact { content: ... }.
as_str() -> Option<&str> Direct string access without JSON parsing. Returns None for Blob.
PartialEq<&str> fact.content == "synthesis complete" works naturally.
Display format!("{}", fact.content) prints the text or <blob: N bytes>.
No From<serde_json::Value> Prevents accidental JSON re-introduction in core. API boundary handles the conversion.

Properties: HashMap<String, Content>

NodeWeight.properties and EdgeWeight.properties changed from HashMap<String, serde_json::Value> to HashMap<String, Content>.

Before:

m.insert("worker".into(), agent.to_string().into());
// postcard cannot serialize serde_json::Value

After:

m.insert("worker".into(), Content::Text(agent.to_string()));
// postcard: (tag=0, len=7, "agent-x")

All property access patterns changed accordingly:

Before After
w.properties.get("key").and_then(v.as_str()) w.properties.get("key").and_then(c.as_str())
w.properties.get("key").and_then(v.as_i64()) w.properties.get("key").and_then(c.as_str()).and_then(s.parse().ok())
w.properties.get("concluded").and_then(v.as_bool()) w.properties.get("concluded").and_then(c.as_str()).is_some_and(v == "true")

Numeric and boolean properties are stored as their string representations. Parsing is done at read time. This is acceptable because properties are small metadata values, not hot-path query columns.

Snapshot serialization: postcard

StorageSnapshot::to_bytes / from_bytes:

Before:

serde_json::to_vec(&snapshot)  // JSON, ~3x larger

After:

postcard::to_allocvec(&self)   // postcard, compact binary
  • Version field added for forward migration.
  • Custom serde module for properties (previous approach with HashMap<String, Value>) is no longer needed.

Flush archive: postcard binary

read_delta_since returned Vec<String> (JSON lines). Changed to Vec<Vec<u8>> (postcard blobs).

Before:

let line = serde_json::to_string(&Fact{...})?;
cold.write_blob(&key, line.as_bytes())?;       // .jsonl file

After:

let bytes = postcard::to_allocvec(&Fact{...})?;
cold.write_blob(&key, &bytes)?;                 // .bin file
  • DeltaSet type alias: (Vec<Vec<u8>>, Vec<Vec<u8>>, Vec<Vec<u8>>).
  • CompositeColdStorage::read_flushed_facts uses postcard::from_bytes::<Fact>(&data) instead of serde_json::from_str::<Fact>(line).
  • Eviction key matching: .jsonl → .bin.
  • No JSON encoding/decoding in the storage layer at all.

conclude_intent: &str instead of &serde_json::Value

fn conclude_intent(&self, intent_id: &str, result: &str) -> Result<Fact, ...>;

The result string is stored as Content::Text(result.to_string()). If the caller wants to store JSON, they serialize it before calling:

bb.conclude_intent("i001", &serde_json::to_string(&json!({...})).unwrap())?;

CypherCapable: moved to nexus-graph

CypherCapable trait was defined in nexus-model/src/storage/cypher.rs with query_plan(&Value) -> Result<Value, String>. Moved to core/graph/src/query/cypher/capable.rs.

  • ColdStorage and HotStorage no longer require CypherCapable — storage backends don’t need to implement query planning.
  • CompositeColdStorage’s CypherCapable impl is a no-op — queries go to petgraph hot storage.
  • DefaultBlackboard holds a separate cold_cypher: Box<dyn CypherCapable> field for DuckDB routing.

cold_query types: moved to nexus-graph

ColdQuery, ColdFilter, etc. were in nexus-model/src/cold_query.rs (DuckDB-specific query types). Moved to core/graph/src/query/cypher/cold_query.rs.

  • ext/storage-duckdb/ imports from nexus_graph::cold_query.
  • Removed pub mod cold_query; from nexus-model.

DetectionCapable: Content instead of serde_json::Value

// Before
fn snapshot_state(&self) -> Option<serde_json::Value> { None }
fn restore_state(&mut self, _state: serde_json::Value) {}
pub type TaskStates = HashMap<String, serde_json::Value>;

// After
fn snapshot_state(&self) -> Option<Content> { None }
fn restore_state(&mut self, _state: Content) {}
pub type TaskStates = HashMap<String, Content>;

Detectors serialize their state as Content::Text(json.to_string()). StorageSnapshot serializes TaskStates via postcard.

Key architectural insight

The original architecture stored graph data in KV (Workers KV) for “recent buffer, fast single-key write”. This meant CompositeColdStorage implemented FactCapable/IntentCapable/HintCapable — the full FIH persistence stack. The KV was the data plane.

The new architecture:

  • Petgraph is the only graph data store (hot storage).
  • CompositeColdStorage manages only durable persistence (blobs, CAS coordination, cursor metadata).
  • DualStorage writes to hot only. Flush extracts deltas from hot and writes postcard blobs to cold.
  • No graph data ever touches KV/MetaStore.

This simplified the storage trait hierarchy considerably:

Verdict

The migration is clean. 82 files changed, ~3,800 lines net removed. All 150+ tests pass. The core crate (nexus-model) no longer depends on serde_json in any way — not in Cargo.toml, not in any source file. Storage crates (nexus-storage-petgraph, nexus-storage-composite) use only postcard for serialization.

The remaining serde_json usage in core/graph (Cypher query engine) and core/process (detector consumer layer) is appropriate — those are higher-level layers that consume the core model, not part of the core model itself.

Future work:

  • Export utility crate: nexus-export-jsonl that reads postcard blobs and writes human-readable output.
  • R2 lifecycle policies for blob eviction (Worker layer, not core).
  • Potentially split core/graph/src/query/cypher/ into its own crate to remove serde_json from core/graph as well.

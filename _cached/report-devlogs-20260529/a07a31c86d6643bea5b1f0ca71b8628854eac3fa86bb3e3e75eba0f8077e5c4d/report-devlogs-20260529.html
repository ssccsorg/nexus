66: IoBuffer Layer and StoreSession — Foundation-Preserving Async Bridge

Date: 2026-05-29

Branch: [66-iobuf-store-session]

Motivation

CF Workers KV/R2/Durable Object bindings are async (Promise-based). The core storage traits (KeyValueStore, BlobStore, ObjectStore) and CompositeColdStorage are sync. An earlier approach attempted to convert all traits to async fn in trait (AFIT), but this was abandoned because it would have propagated async to the entire foundation. During code review it became clear that the foundation must remain sync — not only for WASM compatibility, but because the sync orchestration is a deterministic state transition that multiple consumers (CF Workers, blockchain validators, distributed transaction coordinators) depend on.

Decision: Session Layer over Foundation

Instead of changing the foundation, we introduced an I/O buffer layer between async bindings and sync storage.

Architecture

CF KV / R2 / DO  (source of truth, async)
     ↑ drain_dirty    ↓ hydrate
IoBufferSession
  ├── IoBufferKv      (impl KeyValueStore, dirty tracking)
  ├── IoBufferBlob    (impl BlobStore, dirty tracking)
  ├── IoBufferObject  (impl ObjectStore, dirty tracking)
  └── CompositeColdStorage (sync, unchanged)

Design Patterns

The session traits follow the same fine-grained capability + blanket-impl aggregate pattern used by StorageRead, FactCapable, etc.:

SessionExecute                   SessionDrain
  └── storage() -> &ColdStorage    ├── SessionDrainKv
                                   ├── SessionDrainBlob
                                   └── SessionDrainObject

StoreSession = SessionExecute + SessionDrain
impl<T: SessionExecute + SessionDrain> StoreSession for T {}

This means any future backend (blockchain validator, distributed tx coordinator) can implement StoreSession independently.

Key Insights from Testing

  1. Dirty tracking precision: hydrate_batch() does not mark dirty; set()/delete() do. The consumer must use hydrate_* for preload and drain_* for flush delta.

  2. Local CAS vs remote arbitrage: IoBufferObject.put_state() is a local HashMap CAS. Multiple workers can succeed locally; the remote Durable Object is the final arbiter at flush time. The session layer tracks CAS dirty entries so the consumer can detect rollback.

  3. Atomicity of conclude: conclude_intent produces both a KV delete (intent key) and a KV put (new concluded fact). The consumer must drain both channels and flush in correct order.

  4. Flush_since produces its own dirty: flush_since writes to blob storage and persists a cursor to KV — both tracked as dirty. The consumer must handle this (either drain-clean after flush or accept that flush output is re-flushable).

Remaining Issues

Issues discovered during the PR that are out of scope for this change but must be tracked for future work.

Issue A: flush_since Self-Referential Dirty (Foundation)

Severity: Medium. Scope: IoBuffer* or CompositeColdStorage.

flush_since calls self.blob.put() and self.kv.set() (cursor persist) during iteration. Both write through the IoBuffer* layer, marking the flush output (blob entries + cursor key) as dirty. This means the consumer cannot blindly flush all dirty after execution — flush output must be excluded from the next flush cycle.

Possible resolutions:

  1. Add mark_clean(key) to IoBuffer*, called by consumer after flush.
  2. Add a separate “commit channel” to CompositeColdStorage that bypasses dirty tracking.
  3. Consumer-side: drain-clean immediately after flush, treating flush output as non-replicable metadata.

This also affects blockchain and distributed tx scenarios where flush is an atomic commit and the output (blob + cursor) must be committed atomically.

Status: Unresolved. Consumer (rs-worker) must drain-clean after flush.

Issue B: SessionHandle Deadlock Risk (API)

Severity: Low. Scope: session_server.rs API documentation.

SessionHandle::submit() blocks on recv(). It works correctly when the SessionServer runs on a dedicated thread (spawn()). On WASM where process_one() must be called manually, calling submit() from the same task without concurrent process_one() causes deadlock.

Currently the SessionServer::new() still returns a SessionHandle, but only spawn() should be used in production. The new() handle exists for compatibility.

Status: Mitigated by for_wasm() constructor which returns no handle. Resolved in documentation: tests use spawn() only.

Issue C: IoBuffer* Lock Poisoning (Robustness)

Severity: Low. Scope: iobuf.rs.

All read().unwrap() and write().unwrap() calls panic on poisoned locks. This is consistent with the existing MockKv/MockBlob/MockObject pattern, but IoBuffer* is production code (not test mock). A poisoned lock in a CF Worker would terminate the request.

In practice, RwLock poison only happens when another thread panics while holding the lock. Since IoBuffer* is accessed through a single-threaded SessionServer or via Arc across controlled threads, poisoning risk is low.

Status: Accepted. Same pattern as existing Mock* impls. Future improvement: replace unwrap() with expect() or error propagation.

Issue D: flush_since and conclude_intent Atomicity (Consumer)

Severity: Medium. Scope: Consumer (rs-worker, blockchain adapter).

Both operations produce multiple dirty entries across different channels:

  • conclude_intent: KV delete + KV put (delete intent key, put concluded fact)
  • flush_since: Blob puts + KV set (write blob archive, persist cursor)

The consumer must flush these atomically. For CF Workers this means:

  1. Apply KV deletes first (remove intent key)
  2. Apply KV puts second (write concluded fact)
  3. Apply blob puts last (write flush archive)

If the consumer crashes mid-flush, partial state is observable but not corrupt — the original intent entry still exists and the concluded fact is durable once flushed.

Status: Consumer responsibility. Documented in tests (see test_conclude_intent_delete_and_put_drained_together + test_flush_delta_drain_order_respected_across_channels).

Issue E: ColdStorage Documentation Inconsistency (Docs)

Severity: Medium. Scope: Architecture document (impl_init.qmd).

The impl_init.qmd file had ColdStorage = FihPersistence + FilterCapable, omitting ScanCapable, TimeRangeCapable, FlushCapable, EvictCapable, and CypherCapable. This would cause any reader to misunderstand the storage architecture.

Status: Resolved. Corrected in impl_init.qmd commit 601efec.

Issue F: bindings/cf/ Directory Does Not Exist (Docs)

Severity: Low. Scope: Architecture document (impl_init.qmd).

The Platform Bindings table referenced bindings/cf/ as the CF Worker crate, but no such directory exists. The actual CF Worker binding is planned as rs-worker (separate crate).

Status: Resolved. Updated to gateway/ (planned: rs-worker).

Issue G: Multi-Worker CAS Optimistic Concurrency (Foundation)

Severity: Low. Scope: IoBufferObject.

Two Workers can both succeed at local CAS on the same intent. Only at flush time does the remote Durable Object arbitrate. This is by design — the IoBuffer is an optimistic working copy, not the source of truth. The session layer tracks CAS dirty entries so the consumer can detect which CAS operations succeeded and which were rejected by the remote DO.

Testing confirms: test_cf_worker_cas_rollback_simulation validates that both workers produce dirty CAS entries and the consumer can detect the conflict. test_two_workers_independent_dirty_sets validates that non-overlapping intents produce disjoint dirty sets.

Status: Validated. No change needed.

Issue H: SessionHandle::new() Rediscovery (Code Review)

Severity: Low. Scope: session_server.rs.

SessionServer::new() returns (Self, SessionHandle<S>) but the handle is only usable when the server runs on a separate thread. The for_wasm() constructor correctly returns no handle. The spawn() is the only intended entry point for handle-based usage.

Status: Accepted. The new() handle is harmless because submit() will block forever rather than produce incorrect results. Clearer API separation is future work.

SSCCS Recursion

nexus-core is an implementation of the SSCCS paradigm (Segment, Scheme, Field, Observation, Projection). As the system is applied to broader domains (on-chain data analysis, time-series forecasting, investment intelligence), SSCCS itself becomes a sub-domain within a larger Scheme. This recursive application is the core philosophical validation of the SSCCS model.

Fact is universal. The Fact type (id, origin, content: serde_json::Value, creator) can represent any data domain: on-chain events, time-series embeddings, code review results, sentiment scores. The core is already open to all domains; only the consuming layer needs to define the mapping.

CQRS Commit Channel for flush_since

A major architectural decision was made to resolve the self-referential dirty problem in flush_since() identified in Issue A. The problem: flush_since() writes blob archives and cursor position via self.blob.put() and self.kv.set(), which pass through IoBuffer* and mark every write as dirty. The flush output becomes dirty — a self-referential cycle that violates the CQRS principle (command/query separation).

Decision: CompositeColdStorage gains a commit channel — a separate KV/Blob pair (commit_kv, commit_blob) with dirty tracking disabled. Each IoBuffer* gets a track_dirty: bool field. flush_since() writes exclusively to the commit channel. Consumer drain never sees flush outputs.

FIH Merkle Chain + Cryptographic Signature (#71)

Problem

FIH data nodes (Fact, Intent, Hint) are not cryptographically verifiable:

  • Fact.creator is a plain String — no signature validation
  • FihHash::chain(a, b, c) is hash concatenation, not a true Merkle tree
  • Fact has no prev_id — no chain to predecessor
  • No test for either Merkle structure or signature verification

Current Fact is missing: pubkey: Vec<u8>, signature: Vec<u8>, prev_id: Option<FihHash>.

Required

Layer Change
Core (core/model) Add pubkey, signature, prev_id to Fact/Intent/Hint. FihHash includes prev_id in hash computation.
Agent/Consumer Signature verification (ed25519-dalek). Merkle proof generation/verification (rs_merkle).
Tests Merkle chain serialization, signature roundtrip, tamper detection.

Open Design Questions

  1. Should core enforce signature validation in submit_fact(), or leave it to the consumer?
  2. Should prev_id be mandatory (all Facts form a single chain) or optional (only when explicitly linked)?
  3. Which signature scheme? Ed25519 (simplest, small, fast) or ECDSA (EVM-compatible for DeSci)?
  4. Should Merkle tree be built by nexus (explicit tree storage in CompositeColdStorage) or by the consumer (Fact-level prev_id only)?
  5. Does Intent need the same fields? Intents are mutable (state transitions), signatures may be less meaningful.

Status

Identified. Design phase. Issue #71 tracks this work. Not yet implemented. Crate candidates: ed25519-dalek, rs_merkle, sha2 (already in deps).

2026-05-28 — CompositeColdStorage: 3-tier orchestrated cold storage (Phase 1 final)

CompositeColdStorage is a platform-independent cold storage backend that implements the full ColdStorage trait by orchestrating three independent storage tiers:

  • Tier 1 (KV) — KeyValueStore: fast single-key read/write, recent buffer, cursor persistence
  • Tier 2 (Blob) — BlobStore: JSON-lines archive, bulk scan, flush target
  • Tier 3 (Object) — ObjectStore: CAS-based coordination via get_state/put_state

Architecture

                 ColdStorage trait
                        |
              CompositeColdStorage<K, B, O, C>
                    /       |        \
                KV        Blob      Object
              (recent)   (archive)   (CAS coord.)
                    \       |        /
              Workers KV   R2    Durable Object
              Sled       FS       Redis lock
              MockKv    MockBlob   MockObject

Each tier is a trait that external WASM bindings implement. CompositeColdStorage itself contains zero platform-specific code.

Data flow (final)

  1. Ingest: submit_fact writes to KV only (Tier 1).
  2. Claim: claim_intent atomically gates on object.put_state(key, "", agent). Only one worker passes the CAS. Then updates KV for consistency. Failed CAS returns Conflict.
  3. Flush: flush_since iterates KV keys one by one, filters by submission timestamp (Stamped submitted_at), serializes matching entries to JSON-lines inline, writes to Blob (Tier 2). No intermediate Vec allocation — streaming by design.
  4. Read: read_state reads KV only (recent). scan_partition merges KV (recent) + Blob (flushed) with dedup by entity ID. Partial/corrupt blobs trigger log::warn! with blob key and error detail.
  5. Evict: evict_before on DualStorage delegates to both hot (petgraph) and cold (composite). Composite evicts Blob entries by timestamp. Hot evicts stale intents. Both results summed.
  6. Restart: create_blackboard_from_snapshot_with_cold(snapshot, cold) restores hot petgraph, flush cursor, and claims while accepting an external cold backend (fresh Composite after WASM restart).

Key interface decisions

ObjectStore::put_state(key, expected, new) — a single CAS method covering all coordination patterns:

Call Meaning
put_state(k, "", agent) First claim (expected is empty → key does not exist)
put_state(k, old, new) Ownership transfer
put_state(k, val, "") Release (empty new → delete key)

get_state(key) is the only read path. No set_state — no unconditional write, all writes are CAS.

WASM Worker restart canonical pattern

// 1. Load snapshot from R2
let snapshot: StorageSnapshot = r2.get("snapshot:project").await?;

// 2. Fresh WASM bindings
let kv: KeyValueStore = Namespace::new(env, "KV")?;
let blob: BlobStore = R2Bucket::new(env, "R2")?;
let object: ObjectStore = DurableObject::new(env, "OBJECT")?;

// 3. Composite Cold + Petgraph Hot from snapshot
let cold = CompositeColdStorage::new(kv, blob, object, JsDateClock, "project");
let bb = create_blackboard_from_snapshot_with_cold(snapshot, Box::new(cold));

// 4. Resume incremental flush
bb.flush_since(&bb.to_snapshot().flush_cursor)?;

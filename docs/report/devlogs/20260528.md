# 2026-05-28 — CompositeColdStorage: 3-tier orchestrated cold storage

CompositeColdStorage is a platform-independent cold storage backend that implements the full `ColdStorage` trait (FihPersistence + FilterCapable + ScanCapable + TimeRangeCapable + FlushCapable + CypherCapable) by orchestrating three independent storage tiers:

- **Tier 1 (KV)**: fast single-key read/write, recent buffer, cursor persistence
- **Tier 2 (Blob)**: JSON-lines archive, bulk scan, flush target
- **Tier 3 (Object)**: CAS-based coordination, snapshot ownership (Durable Object future)

## Architecture

```
                 ColdStorage trait
                        |
              CompositeColdStorage<K, B, O, C>
                    /       |        \
                KV Tier   Blob Tier   Object Tier
              (recent)   (archive)    (coordination)
                    \       |        /
              Workers KV    R2     Durable Object
              MockKv      MockBlob   MockObject
```

Each tier is a trait (`KeyValueStore`, `BlobStore`, `ObjectStore`) that external WASM bindings implement. `CompositeColdStorage` itself contains zero platform-specific code.

## Data flow

1. **Ingest**: `submit_fact` writes to KV only (Tier 1).
2. **Flush**: `flush_since` reads KV entries newer than cursor, serializes to JSON-lines, writes to Blob (Tier 2). Cursor persisted in KV.
3. **Read**: `read_state` reads KV only (recent). `scan_partition` merges KV (recent) + Blob (flushed) with dedup by entity ID.
4. **Evict**: `evict_before` removes old Blob entries by timestamp. KV entries are never evicted by this path.
5. **Coordinate** (future): `ObjectStore.compare_and_swap` in `claim_intent` for cross-worker atomic claim.

### Key invariants established

- **DualStorage** delegates writes to both hot (Petgraph) and cold (Composite). Reads from hot.
- **Flush** copies hot data to cold; hot data is never removed. `flush_since` is incremental via cursor.
- **Snapshot** captures only the hot petgraph state. Cold storage is reconstructed fresh after restore. The `flush_cursor` survives in the snapshot.
- **R2/blob** provides data durability across worker restarts. Composite KV is lost, but archived data in Blob is retained.
- **NullStorage** is a no-op cold backend, not a bug workaround. It exists because `DualStorage` always requires a cold slot.

### Extreme scenarios assessed

| Scenario | Verdict | Mechanism |
|----------|---------|-----------|
| Worker crash during flush | Data safe, possible dedup | Cursor not advanced → re-flush, dedup_by handles duplicates |
| Worker restart before any flush | Data safe | Hot petgraph in snapshot, restored on restart |
| Multi-lifetime data preservation | Tested and verified | R2 retains all flushed data across restarts |
| WASM memory limit during flush | Not addressed (Phase 2) | `flush_since` loads all KV data to memory |
| Blob accumulation without eviction | Not addressed (Phase 2) | Explicit `evict_before` needed |
| Concurrent claim race with KV-only | Documented, not fixed | `ObjectStore.compare_and_swap` designed for this |

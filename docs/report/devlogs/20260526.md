## Summary

Implemented FlushCursor persistence across scheduler invocations for
incremental flush, resolved DuckDB read_parquet empty glob issue via
sentinel Parquet files, expanded the flush test suite to 17 tests
covering concurrent, rapid, and read-during-flush scenarios, and
added partition-scoped view refresh for project isolation.

## Changes

### FlushCursor Persistence (#58)

- Added `Serialize` / `Deserialize` to `FlushCursor` so it can be stored
  in `StorageSnapshot` alongside graph, claims, and task_states.
- Added `flush_cursor: FlushCursor` field to `StorageSnapshot` with
  `#[serde(default)]` for backward compatibility with snapshots that
  predate cursor tracking.
- Implemented `read_flush_cursor()` / `update_flush_cursor()` in
  `DefaultBlackboard` using hot graph metadata nodes
  (`__flush_cursor_{project_id}`).
- `DefaultBlackboard::flush(&mut self)` now reads the persisted cursor,
  passes it to `storage.flush_since(&cursor)`, and persists the returned
  `new_cursor` back to the hot graph.
- `from_snapshot_inner()` restores the flush cursor as a metadata node
  so cursor state survives worker restart.

### DuckDbStorage Flush Implementation

- `DuckDbStorage::flush_since()` exports facts, intents, and hints
  newer than `cursor.last_flushed_at` to Parquet files via
  `COPY ... TO (FORMAT PARQUET)`.
- Output is partitioned by `{base_path}/{entity}/partition={project_id}/`.
- Views are refreshed after flush so subsequent reads include the new
  Parquet files.
- The WHERE clause `created_at > cursor.last_flushed_at` uses DuckDB's
  native string comparison; sentinel files use explicit `::VARCHAR` cast
  to prevent type inference issues.

### Sentinel Parquet Files

DuckDB's `read_parquet()` throws `IOException` when the glob pattern
matches zero files. This caused `CREATE VIEW IF NOT EXISTS` to fail on
first run with empty directories. Resolution: `DuckDbStorage::new()`
writes a 0-row sentinel Parquet file per entity type (`.sentinel.parquet`).
Sentinel files satisfy the "at least one file" glob requirement without
affecting query results or flush counts.

### Process Layer Wiring

- `try_evict_flush()` signature changed: now accepts `&FlushCursor` and
  returns `Result<(u64, FlushCursor), String>`. The cursor is passed
  through rather than hardcoded to empty.
- `Scheduler::_tick_inner()` evict_fn signature changed from `&B` to
  `&mut B` to allow `DefaultBlackboard::flush(&mut self)`.
- `Scheduler::tick_with_flush()` constructs an empty cursor (full flush)
  at scheduler level; incremental cursor persistence is delegated to the
  gateway or higher-level orchestrator.

### FlushResult Convenience

- Added `From<FlushResult> for (u64, FlushCursor)` so callers can
  destructure with `let (count, cursor) = result.into()`.

## DuckDB Source Analysis

Inspected DuckDB source at `duckdb/src/` to understand three issues:

1. **Empty glob behavior**: `read_parquet()` with `FALLBACK_GLOB` throws
   `IOException` when no files match (`file_system.cpp#L688-702`).
   Resolution: sentinel Parquet files.

2. **String comparison**: DuckDB uses byte-by-byte `memcmp` for VARCHAR
   comparison (`string_type.hpp#L233-240`). Lexicographic ordering means
   `'100' > '50'` is false. Epoch-second timestamps compare correctly
   when both sides use the same format.

3. **COPY ... TO with view**: `PhysicalCopyToFile` evaluates the WHERE
   clause via standard query planning. Filtering happens in child
   operators before the COPY sink receives data.

## Architectural Insight: composite as Core Cold Storage

Current DuckDB cold storage works well for server environments but
cannot run in WASM/CF Workers (C bindings). The long-term architecture
separates platform-specific storage from core:

| Layer | Directory | Description |
|-------|-----------|-------------|
| Core storage | `core/storage/composite/` | Platform-independent KV + Blob traits |
| Core cold impl | `core/storage/composite/` | `parquet-wasm` for Parquet I/O |
| Server ext | `ext/storage-duckdb/` | DuckDB cold storage (C bindings, non-WASM) |

The `composite` module introduces two traits:
- `KeyValueStore`: get/put/list for recent Fact/Intent/Hint + cursor
- `BlobStore`: put_blob/get_blob for Parquet chunks (R2, S3, local fs)

This allows the same `FlushCapable` / `ColdStorage` trait interface to
run on CF Workers (KV + R2 + parquet-wasm) or dedicated servers (DuckDB).

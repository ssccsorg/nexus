# 135-nexd: Unified native daemon for the nex ecosystem

## Context

`nexd` is the unified daemon for the nex ecosystem, acting as the operating system for a swarm of autonomous agents. It provides FIH blackboard (shared memory), process management, Unix socket IPC, and lifecycle management for `nex-*` applications. The conceptual reference is that `nexd` is to `nex-*` as iOS is to apps: it provides foundational services that agents consume through a well-defined interface.

The daemon is built on the `proc-daemon` framework for graceful shutdown, signal handling, and subsystem lifecycle management. It reuses `nex::create_blackboard()` to produce a `HybridBlackboard` instance, following the same pattern established in `nex-api`.

## Key technical decisions

### proc-daemon as foundation (v1.1.2)

The `proc-daemon` crate provides a battle-tested subsystem lifecycle: each subsystem runs as a concurrent task with typed channels for shutdown signals. The daemon initializes subsystems in dependency order and drains them in reverse on SIGTERM/SIGINT.

**Pros**: Production-ready graceful shutdown, signal handling, subsystem management out of the box.

**Cons**: Approximately 70 transitive dependencies pulled in. The tracing initialization in `proc-daemon` can conflict with the caller's own tracing subscriber, requiring `try_init` instead of `init`.

### HybridBlackboard reuse

Rather than duplicating blackboard logic, `nexd` consumes `nex::create_blackboard()` directly and wraps the returned `HybridBlackboard` in `Arc<Mutex<HybridBlackboard>>`. This preserves access to the `EvictCapable` trait while providing interior mutability.

### Process Manager in Phase 1

Process management was implemented from the start rather than deferred. The `ProcessManager` handles:

- **spawn**: Fork-exec of `nex-*` agent binaries with environment injection
- **reap**: Periodic collection of exited child processes via `waitpid` with `WNOHANG`
- **kill**: SIGTERM sent to agent processes, with escalation to SIGKILL after timeout
- **sync shutdown**: The shutdown routine collects all child PIDs, sends SIGTERM, and waits for graceful exit, avoiding the pitfall of holding a `MutexGuard` across `.await` points

### Scheduler uses try_lock

The OODA-loop scheduler acquires the blackboard lock via `try_lock()` instead of `lock()`. This prevents blocking IPC handlers that may be awaiting the same lock during a long-running OODA tick. If the lock cannot be acquired immediately, the tick is skipped.

### Connection semaphore

A maximum of 128 concurrent Unix socket connections is enforced via a `tokio::sync::Semaphore`. This prevents resource exhaustion from unbounded connection spawning under DoS or high load.

### Read methods added after review

The initial implementation exposed only `read_state` for reading blackboard entries. After code review, `read_fact`, `read_intent`, and `read_hint` were added to provide symmetric read access for each FIH slot type.

## Code review outcomes

Five inline comments were raised during PR review:

| Comment | Resolution |
|---------|-----------|
| Sync handler under async dispatch | Made the dispatch function sync — the IPC handlers are synchronous and should not be wrapped in `spawn` |
| `write_hint` error mapping inconsistency | Fixed with `map_error()` to produce uniform error types across all write methods |
| Unbounded connection spawn | Wrapped accept loop in `Semaphore::acquire` to limit concurrent connections to 128 |
| Scheduler lock contention | Changed `lock()` to `try_lock()` with tick skip on contention |
| Agent spawn error silent | Added `error!()` logging on agent spawn failure |

Four comments were resolved immediately. The single-read methods (`read_fact`, `read_intent`, `read_hint`) were resolved in a follow-up commit.

## Testing

### Integration tests (27 tests)

The test suite covers three categories:

- **Transport** (connection, pipelining, concurrent connections, large messages, partial writes): Validates that the Unix socket transport layer handles framing, backpressure, and concurrent clients correctly.
- **Communication** (agent-to-agent via blackboard, delegation pattern, shared blackboard): Verifies that agents can read and write FIH slots through `nexd` and that delegation across agents works.
- **Lifecycle** (spawn/kill, multi-agent, reap, graceful shutdown): Ensures that the process manager correctly spawns agents, reaps zombies, and shuts down cleanly.

### verify_nexd in run.sh

Seven real-world scenarios are executed in CI:

1. FIH operations (write + read round-trip for facts, intents, hints, state)
2. Intent lifecycle (claim → update → conclude sequence)
3. Agent lifecycle (spawn agent, verify it responds, kill it)
4. Error handling (double-claim detection, wrong-agent rejection, conclude without claim)
5. Graceful shutdown (SIGTERM delivery, reaping, clean exit code)
6. Concurrent operations (multiple clients writing and reading simultaneously)
7. Read by ID (targeted fact/intent/hint retrieval by identifier)

## Gaps discovered

**Claim-before-conclude not enforced by HybridBlackboard**: The underlying blackboard implementation allows concluding an intent that was never claimed. This is a semantic gap that should be enforced at the `nexd` IPC layer.

**Short-lived process reap delay**: The reaper runs on a 5-second `try_reap` interval. A short-lived agent that exits and is reaped within the same tick window may still show as running in the process table for up to 5 seconds.

**No push/subscribe mechanism**: Agents can only poll for changes. There is no event subscription model where an agent registers a callback for blackboard mutations. This limits responsiveness in event-driven workflows.

**Single read methods added post-review**: The initial PR shipped without `read_fact`, `read_intent`, or `read_hint`. These were identified as missing during review and added as a follow-up.

**proc-daemon tracing conflict**: The `proc-daemon` crate initializes its own tracing subscriber. When `nexd` is called from a parent process that also sets up tracing (e.g., during integration tests), the second `init` panics. The workaround is `tracing_subscriber::reload::try_init`.

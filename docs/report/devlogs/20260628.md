# 134-nex-spinwasi-ssccsdocs: From WASIX to Spin WASI

## Context

PR #134 aimed to separate all apps from the core workspace into independent Cargo projects. One of those apps was `nex-wasmer-ssccsdocs` — a document ingestion and BM25 semantic search service deployed on Wasmer Edge. The app uses FIH Blackboard as the storage substrate and syncs SSCCS documentation from `docs.ssccs.org` into Facts for semantic search.

## The WASIX dead end

### Attempt: Wasmer Edge WASIX Proxy

The original implementation used the WASIX toolchain (`rustc 1.90.0-dev`, the `wasix` Rustup toolchain) to compile for `wasm32-wasmer-wasi` target. Wasmer Edge's Proxy mode works by having the WASM module start an internal HTTP server (axum + tokio) on a configurable port, which the Edge gateway then proxies external requests to.

**Result**: All 27 deployed versions (v1 through v27) returned `workload_failure` → HTTP 500.

### Root cause

The `wasix` toolchain always produces WASM binaries with:

- 90 `wasix_32v1` imports (thread spawn, futex, epoll, sock_accept_v2, etc.)
- `env.memory` with `shared: true` (shared memory for threading)
- `wasi.thread-spawn` import

Neither the local Wasmer 7.1.0 CLI nor Wasmer Edge could run these binaries. Local validation failed with:

```text
No backends support the required features for the Wasm module.
```

The official Wasmer Edge documentation confirms that TCP/UDP applications (which Proxy mode relies on via WASIX sockets) are still in alpha phase:

> "Although TCP and UDP applications are supported in Wasmer Edge, we haven't yet exposed the API as it's still in alpha phase."

For HTTP workloads, the documentation recommends WCGI mode (CGI over stdin/stdout) but that would require rewriting the HTTP handler layer.

## The Spin pivot

### Decision: Fermyon Spin + standard WASI

Fermyon Spin provides a higher-level HTTP abstraction: instead of the WASM module managing its own TCP listener, Spin's runtime handles HTTP and passes requests to the component via the WASI component model. This eliminates the need for WASIX entirely.

Target: `wasm32-wasip2` (WASI preview 2 / Component Model) using the standard stable Rust toolchain — no forks, no patches.

### Phase 1: Spin 4 SDK (`spin-sdk = "6.0.0"`)

Initial migration used Spin 4's `#[http_service]` macro and `spin_sdk::http::Request`/`Response` types.

**Result**: Local `spin build --up` worked perfectly. All endpoints functional.

**Hurdle**: Fermyon Cloud returned 404 for all routes. Even a minimal "Hello!" app failed with `failed to prepare your Spin application`.

**Root cause**: Fermyon Cloud runs Spin 3.x which does not support the WASI preview 3 ABI produced by spin-sdk 6.x.

### Phase 2: Spin 3 SDK (`spin-sdk = "5.2.0"`)

Downgraded to spin-sdk 5.2.0 which uses `#[http_component]` macro and standard `http::Request`/`http::Response` types (from the `http` crate).

Key changes:

- `#[http_service]` → `#[http_component]`
- `spin_sdk::http::Request` → `http::Request<Vec<u8>>`
- `Response::builder().body(text)?` with `?` operator
- `spin_sdk::http::send(req)` for outbound HTTP (replacing `spin_sdk::http::get`)
- Removed `mem_io.rs`, `batch_io.rs`, `wasmer_io.rs` — replaced with in-memory `MemIo`

**Result**: Deployed successfully on Fermyon Cloud. All endpoints return 200.

### Key design decisions

| Decision | Rationale |
|----------|-----------|
| `MemIo` over `FsIo` | WASI sandbox restricts filesystem writes. In-memory HashMap avoids permission issues entirely. |
| `OnceLock` + `AtomicBool` lazy init | Storage initializes on first request. Doc sync runs once per cold start (cheap via delta sync). |
| `#[http_component]` over `#[http_service]` | Spin 3.x compatibility for Fermyon Cloud. |
| Manual URL routing | Avoids axum dependency. 15 routes matched by method + path pattern. |
| standard `http` crate types | `spin_sdk::http::Request` differs between SDK versions. `http::Request<Vec<u8>>` is stable across versions. |


## Lessons learned

1. **WASIX is not production-ready on Wasmer Edge.** The `wasix_32v1` ABI and shared memory threading model are incompatible with both local Wasmer 7.1.0 and the Edge platform. The docs explicitly state TCP/UDP apps are alpha.

2. **Fermyon Cloud lags behind Spin CLI.** Spin 4 (SDK 6.x, WASI preview 3) is not supported. Must use Spin 3 (SDK 5.x, `#[http_component]`, manifest v2 targeting wasm32-wasip2).

3. **Platform abstraction pays off.** The entire business logic (`FihStorage`, `InMemoryBm25`, `ingest_document`, BM25 search) moved from WASIX/axum to Spin with zero changes. Only the HTTP handler layer and IO backend needed migration.

4. **In-memory storage is sufficient for stateless workloads.** Data persists per WASM instance. Cross-instance persistence requires Spin's key-value store (future work).

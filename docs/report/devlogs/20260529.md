# 66: IoBuffer Layer and StoreSession — Foundation-Preserving Async Bridge

**Date**: 2026-05-29

**Branch**: [66-iobuf-store-session]

## Motivation

CF Workers KV/R2/Durable Object bindings are async (Promise-based). The core
storage traits (`KeyValueStore`, `BlobStore`, `ObjectStore`) and
`CompositeColdStorage` are sync. An earlier approach attempted to convert all
traits to `async fn in trait` (AFIT), but this was abandoned because it would
have propagated async to the entire foundation. During code review it became
clear that the foundation must remain sync — not only for WASM compatibility,
but because the sync orchestration is a deterministic state transition that
multiple consumers (CF Workers, blockchain validators, distributed transaction
coordinators) depend on.

## Decision: Session Layer over Foundation

Instead of changing the foundation, we introduced an I/O buffer layer between
async bindings and sync storage.

### Architecture

```text
CF KV / R2 / DO  (source of truth, async)
     ↑ drain_dirty    ↓ hydrate
IoBufferSession
  ├── IoBufferKv      (impl KeyValueStore, dirty tracking)
  ├── IoBufferBlob    (impl BlobStore, dirty tracking)
  ├── IoBufferObject  (impl ObjectStore, dirty tracking)
  └── CompositeColdStorage (sync, unchanged)
```

### Design Patterns

The session traits follow the same fine-grained capability + blanket-impl
aggregate pattern used by `StorageRead`, `FactCapable`, etc.:

```text
SessionExecute                   SessionDrain
  └── storage() -> &ColdStorage    ├── SessionDrainKv
                                   ├── SessionDrainBlob
                                   └── SessionDrainObject

StoreSession = SessionExecute + SessionDrain
impl<T: SessionExecute + SessionDrain> StoreSession for T {}
```

This means any future backend (blockchain validator, distributed tx
coordinator) can implement `StoreSession` independently.

### Key Insights from Testing

1. **Dirty tracking precision**: `hydrate_batch()` does not mark dirty;
   `set()`/`delete()` do. The consumer must use `hydrate_*` for preload
   and `drain_*` for flush delta.

2. **Local CAS vs remote arbitrage**: `IoBufferObject.put_state()` is a
   local HashMap CAS. Multiple workers can succeed locally; the remote
   Durable Object is the final arbiter at flush time. The session layer
   tracks CAS dirty entries so the consumer can detect rollback.

3. **Atomicity of conclude**: `conclude_intent` produces both a KV delete
   (intent key) and a KV put (new concluded fact). The consumer must
   drain both channels and flush in correct order.

4. **Flush_since produces its own dirty**: `flush_since` writes to blob
   storage and persists a cursor to KV — both tracked as dirty. The
   consumer must handle this (either drain-clean after flush or accept
   that flush output is re-flushable).

## Remaining Issues

Issues discovered during the PR that are out of scope for this change but must
be tracked for future work.

### Issue A: `flush_since` Self-Referential Dirty (Foundation)

**Severity**: Medium. **Scope**: IoBuffer* or CompositeColdStorage.

`flush_since` calls `self.blob.put()` and `self.kv.set()` (cursor persist)
during iteration. Both write through the IoBuffer* layer, marking the flush
output (blob entries + cursor key) as dirty. This means the consumer cannot
blindly flush all dirty after execution — flush output must be excluded from
the next flush cycle.

Possible resolutions:

1. Add `mark_clean(key)` to IoBuffer*, called by consumer after flush.
2. Add a separate "commit channel" to CompositeColdStorage that bypasses
   dirty tracking.
3. Consumer-side: drain-clean immediately after flush, treating flush output
   as non-replicable metadata.

This also affects blockchain and distributed tx scenarios where flush is an
atomic commit and the output (blob + cursor) must be committed atomically.

**Status**: Unresolved. Consumer (rs-worker) must drain-clean after flush.

---

### Issue B: `SessionHandle` Deadlock Risk (API)

**Severity**: Low. **Scope**: session_server.rs API documentation.

`SessionHandle::submit()` blocks on `recv()`. It works correctly when the
SessionServer runs on a dedicated thread (`spawn()`). On WASM where
`process_one()` must be called manually, calling `submit()` from the same
task without concurrent `process_one()` causes deadlock.

Currently the `SessionServer::new()` still returns a `SessionHandle`, but
only `spawn()` should be used in production. The `new()` handle exists for
compatibility.

**Status**: Mitigated by `for_wasm()` constructor which returns no handle.
Resolved in documentation: tests use `spawn()` only.

---

### Issue C: `IoBuffer*` Lock Poisoning (Robustness)

**Severity**: Low. **Scope**: iobuf.rs.

All `read().unwrap()` and `write().unwrap()` calls panic on poisoned locks.
This is consistent with the existing `MockKv/MockBlob/MockObject` pattern,
but IoBuffer* is production code (not test mock). A poisoned lock in a CF
Worker would terminate the request.

In practice, `RwLock` poison only happens when another thread panics while
holding the lock. Since IoBuffer* is accessed through a single-threaded
SessionServer or via `Arc` across controlled threads, poisoning risk is low.

**Status**: Accepted. Same pattern as existing Mock* impls. Future
improvement: replace `unwrap()` with `expect()` or error propagation.

---

### Issue D: `flush_since` and `conclude_intent` Atomicity (Consumer)

**Severity**: Medium. **Scope**: Consumer (rs-worker, blockchain adapter).

Both operations produce multiple dirty entries across different channels:

- `conclude_intent`: KV delete + KV put (delete intent key, put concluded fact)
- `flush_since`: Blob puts + KV set (write blob archive, persist cursor)

The consumer must flush these atomically. For CF Workers this means:
1. Apply KV deletes first (remove intent key)
2. Apply KV puts second (write concluded fact)
3. Apply blob puts last (write flush archive)

If the consumer crashes mid-flush, partial state is observable but not
corrupt — the original intent entry still exists and the concluded fact is
durable once flushed.

**Status**: Consumer responsibility. Documented in tests (see
`test_conclude_intent_delete_and_put_drained_together` +
`test_flush_delta_drain_order_respected_across_channels`).

---

### Issue E: `ColdStorage` Documentation Inconsistency (Docs)

**Severity**: Medium. **Scope**: Architecture document (`impl_init.qmd`).

The `impl_init.qmd` file had `ColdStorage = FihPersistence + FilterCapable`,
omitting ScanCapable, TimeRangeCapable, FlushCapable, EvictCapable, and
CypherCapable. This would cause any reader to misunderstand the storage
architecture.

**Status**: Resolved. Corrected in `impl_init.qmd` commit 601efec.

### Issue F: `bindings/cf/` Directory Does Not Exist (Docs)

**Severity**: Low. **Scope**: Architecture document (`impl_init.qmd`).

The Platform Bindings table referenced `bindings/cf/` as the CF Worker crate,
but no such directory exists. The actual CF Worker binding is planned as
`rs-worker` (separate crate).

**Status**: Resolved. Updated to `gateway/ (planned: rs-worker)`.

---

### Issue G: Multi-Worker CAS Optimistic Concurrency (Foundation)

**Severity**: Low. **Scope**: IoBufferObject.

Two Workers can both succeed at local CAS on the same intent. Only at flush
time does the remote Durable Object arbitrate. This is by design — the IoBuffer
is an optimistic working copy, not the source of truth. The session layer
tracks CAS dirty entries so the consumer can detect which CAS operations
succeeded and which were rejected by the remote DO.

Testing confirms: `test_cf_worker_cas_rollback_simulation` validates that
both workers produce dirty CAS entries and the consumer can detect the
conflict. `test_two_workers_independent_dirty_sets` validates that
non-overlapping intents produce disjoint dirty sets.

**Status**: Validated. No change needed.

---

### Issue H: `SessionHandle::new()` Rediscovery (Code Review)

**Severity**: Low. **Scope**: session_server.rs.

`SessionServer::new()` returns `(Self, SessionHandle<S>)` but the handle is
only usable when the server runs on a separate thread. The `for_wasm()`
constructor correctly returns no handle. The `spawn()` is the only intended
entry point for handle-based usage.

**Status**: Accepted. The `new()` handle is harmless because `submit()` will
block forever rather than produce incorrect results. Clearer API separation
is future work.

## Nexus-SIM: Virtual Emulation Test Suite

**Status**: Identified. Separate issue.

VECompositeStorage (ext/ve-composite) exposes KeyValueStore, BlobStore, and ObjectStore semantics over async HTTP, simulating the Cloudflare Workers async I/O boundary. Every request goes through IoBufferSession, so dirty tracking is verified at the HTTP boundary — not at the Rust function call level.

### Strategic Importance

The virtual emulation pattern extends far beyond CF Workers:

| Domain | Emulated | Real |
|---|---|---|
| Edge compute | VECompositeStorage HTTP | CF Workers KV/R2/DO |
| Robotics | ROS2 topic pub/sub over simulated nodes | Real ROS2 Humble |
| Blockchain | Simulated validator with StoreSession | On-chain WASM runtime |
| Edge AI | Fake ONNX runtime returning canned results | Real NPU/GPU inference |
| Distributed tx | Local 2PC coordinator with injected failures | Global consensus |

The test suite itself becomes a platform. Each emulated backend implements the same `StoreSession` trait from nexus-model. Switching from emulation to production requires only swapping the binding crate — no core changes.

### Architecture

```
Tests (async HTTP)
  ↓
VECompositeStorage HTTP Server (ext/ve-composite)
  ├── /kv/{project}/{key}   GET/PUT/DELETE
  ├── /r2/{project}/{key}   PUT/GET/DELETE
  ├── /do/{project}/{key}/cas POST
  └── /drain/{project}/kv-puts GET  (dirty inspection)
      ↓ sync
IoBufferSession + CompositeColdStorage
```

### Future Direction: nexus-sim workspace

When the suite grows to cover ROS2, blockchain, distributed tx, and edge AI emulation, it should graduate to a dedicated `nexus-sim` workspace with its own CI pipeline. The `ext/ve-composite` crate is the seed.

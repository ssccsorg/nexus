# 66: IoBuffer Layer and StoreSession — Foundation-Preserving Async Bridge

**Date**: 2026-05-29

**Branch**: [66-iobuf-store-session]

## Motivation

CF Workers KV/R2/Durable Object bindings are async (Promise-based). The core
storage traits (`KeyValueStore`, `BlobStore`, `ObjectStore`) and
`CompositeColdStorage` are sync. An earlier approach attempted to convert all
traits to `async fn in trait` (AFIT), but this was abandoned because it would
have propagated async to the entire foundation. During code review it became
clear that the foundation must remain sync — not only for WASM compatibility,
but because the sync orchestration is a deterministic state transition that
multiple consumers (CF Workers, blockchain validators, distributed transaction
coordinators) depend on.

## Decision: Session Layer over Foundation

Instead of changing the foundation, we introduced an I/O buffer layer between
async bindings and sync storage.

### Architecture

```text
CF KV / R2 / DO  (source of truth, async)
     ↑ drain_dirty    ↓ hydrate
IoBufferSession
  ├── IoBufferKv      (impl KeyValueStore, dirty tracking)
  ├── IoBufferBlob    (impl BlobStore, dirty tracking)
  ├── IoBufferObject  (impl ObjectStore, dirty tracking)
  └── CompositeColdStorage (sync, unchanged)
```

### New Components

| Component | Location | Role |
|---|---|---|
| `IoBufferKv` | `core/storage/composite/src/iobuf.rs` | In-memory KV with dirty puts+deletes tracking |
| `IoBufferBlob` | `core/storage/composite/src/iobuf.rs` | In-memory blob store with dirty tracking |
| `IoBufferObject` | `core/storage/composite/src/iobuf.rs` | In-memory CAS store with dirty tracking |
| `IoBufferSession` | `core/storage/composite/src/store_session.rs` | Owns IoBuffer trio + CompositeColdStorage |
| `SessionServer<S>` | `core/storage/composite/src/session_server.rs` | Generic serialized job queue |
| `SessionHandle<S>` | `core/storage/composite/src/session_server.rs` | Clonable job submit handle (native: spawn, WASM: manual process_one) |
| `StoreSession` trait | `core/model/src/storage/session.rs` | Aggregate trait: SessionExecute + SessionDrain* |
| `SessionExecute` | `core/model/src/storage/session.rs` | Access to ColdStorage for sync ops |
| `SessionDrainKv/Blob/Object` | `core/model/src/storage/session.rs` | Drain dirty entries per channel |

### Design Patterns

The session traits follow the same fine-grained capability + blanket-impl
aggregate pattern used by `StorageRead`, `FactCapable`, etc.:

```text
SessionExecute                   SessionDrain
  └── storage() -> &ColdStorage    ├── SessionDrainKv
                                   ├── SessionDrainBlob
                                   └── SessionDrainObject

StoreSession = SessionExecute + SessionDrain
impl<T: SessionExecute + SessionDrain> StoreSession for T {}
```

This means any future backend (blockchain validator, distributed tx
coordinator) can implement `StoreSession` independently.

### Key Insights from Testing

1. **Dirty tracking precision**: `hydrate_batch()` does not mark dirty;
   `set()`/`delete()` do. The consumer must use `hydrate_*` for preload
   and `drain_*` for flush delta.

2. **Local CAS vs remote arbitrage**: `IoBufferObject.put_state()` is a
   local HashMap CAS. Multiple workers can succeed locally; the remote
   Durable Object is the final arbiter at flush time. The session layer
   tracks CAS dirty entries so the consumer can detect rollback.

3. **Atomicity of conclude**: `conclude_intent` produces both a KV delete
   (intent key) and a KV put (new concluded fact). The consumer must
   drain both channels and flush in correct order.

4. **Flush_since produces its own dirty**: `flush_since` writes to blob
   storage and persists a cursor to KV — both tracked as dirty. The
   consumer must handle this (either drain-clean after flush or accept
   that flush output is re-flushable).

### What Did Not Change

- `KeyValueStore`, `BlobStore`, `ObjectStore` — sync, unchanged
- `CompositeColdStorage` — sync, unchanged
- `ColdStorage`, `HotStorage`, `FihPersistence` — sync, unchanged
- All 30 existing tests — passing, unchanged
- Zero dependencies added (no `async-trait`, no nightly)
